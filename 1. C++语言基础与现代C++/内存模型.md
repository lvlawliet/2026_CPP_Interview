# 内存模型   
## 栈/堆/全局/常量区/代码区  
<pre>
高地址  
┌────────────────────┐  
│       栈区 (Stack) │ ← 向下增长（函数调用）  
├────────────────────┤  
│       堆区 (Heap)  │ ← 向上增长（new/malloc）  
├────────────────────┤  
│ 未初始化数据 (BSS) │  
├────────────────────┤  
│ 已初始化数据 (Data)│  
├────────────────────┤  
│     常量区 (ROData)│  
├────────────────────┤  
│     代码区 (Text)  │ ← 最低地址  
└────────────────────┘  
低地址  
</pre>  
### 代码区（Code Segment / Text Segment）  
存放内容：程序编译后的机器指令（可执行代码）  
特点：  
只读（防止被修改）  
可共享（多个进程可以映射同一份代码，节省内存）  
大小在编译期确定，运行时不可变  
  
生命周期：整个程序运行期间  
示例：你写的 main()、func() 等函数体都在这里  
“代码区存放二进制指令，不允许修改，否则会触发段错误（Segmentation Fault）”  
  
### 常量区（Constant Segment / ROData）  
存放内容：  
字符串字面量（如 "hello world"） 
const全局/静态变量  
constexpr 常量（C++11起）  
  
特点：  
只读（尝试修改会段错误）  
编译期确定  
属于只读数据段，常与代码区放在一起  

生命周期：整个程序  
示例：  
``` cpp
const int GLOBAL_CONST = 100;     // 常量区
constexpr double PI = 3.14;       // 常量区
const char* str = "hello";        // "hello" 在常量区，str指针在栈/全局
```
### 全局区 / 静态存储区（Data Segment + BSS）  
细分为两部分：  
已初始化数据段（Data）：显式初始化过的全局/静态变量  
未初始化数据段（BSS）：未初始化的全局/静态变量（系统自动清0）  
  
存放内容：  
全局变量（int g = 10;）  
静态变量（static int s; 无论在函数内还是外）  
static 局部变量（只初始化一次）  
  
特点：  
程序启动前分配，程序结束时释放  
生命周期 = 整个程序  

示例：  
``` cpp
int global_var = 10;          // Data段
static int static_global = 20;// Data段
int uninit_global;            // BSS段（值为0）

void func() {
    static int count = 0;     // 静态局部变量，只初始化一次，放在全局区
    count++;
}
```

### 堆区（Heap）  
存放内容：程序员动态申请的内存  
分配方式：new / malloc / calloc / realloc
特点：  
向上增长（地址越来越大）  
大小几乎不受限制（受虚拟内存和系统限制）  
手动管理（delete / free）  
分配/释放较慢（涉及系统调用）  
  
生命周期：从 new 到 delete，可跨函数  
示例：    
``` cpp
int* p = new int[1000000];  // 堆
delete[] p;
```   
### 栈区（Stack）  
存放内容：  
局部变量  
函数参数  
返回地址、调用者栈帧（ebp、esp）  
  
分配方式：自动（进入函数时分配，退出时释放）  
特点：  
向下增长（地址越来越小）  
大小有限（Linux默认8MB，Windows默认1MB，可通过编译器参数修改）  
分配/释放极快（只需移动栈指针）  
  
生命周期：函数作用域结束自动释放  
示例：  
``` cpp
void func(int param) {      // param 在栈
    int local = 10;         // local 在栈
    int arr[1000000];       // 太大 → 栈溢出！
}
```  
### 最常被问的10个问题  
栈和堆的区别？  
→ 见上表 + “栈自动管理、连续、快；堆手动、易碎片、慢”  
为什么栈比堆快？  
→ 栈只需移动esp指针；堆需要系统调用 + 找空闲块（可能用伙伴系统/空闲链表）  
const全局变量和const局部变量分别在哪？  
→ 全局const在常量区；局部const通常在栈（除非被优化到常量区）  
static变量在哪？什么时候初始化？  
→ 全局区；程序启动前（全局静态）或第一次调用时（局部静态）  
new 和 malloc 的区别？  
→ new是运算符（调用构造函数）；malloc是函数（只分配内存）  
字符串 "hello" 在哪？  
→ 常量区（只读）  
全局变量未初始化值是多少？  
→ 0（BSS段特性）  
如何防止栈溢出？  
→ 避免大数组、减少递归深度、用堆代替大局部变量  
内存泄漏如何检测？  
→ valgrind、AddressSanitizer、CRT调试堆  
C++11之后新增的内存相关关键字？  
→ thread_local（线程局部存储）、constexpr、std::aligned_storage 等  
  
## new/delete vs malloc/free  
| 对比维度              | new / delete                                      | malloc / free                                      |
|-----------------------|---------------------------------------------------|----------------------------------------------------|
| **本质**              | **运算符**（可重载）                              | **库函数**（不能重载）                             |
| **分配内存**          | 调用 `operator new` 分配 + 调用构造函数           | 只调用底层分配器（通常是 `malloc`）                |
| **释放内存**          | 调用析构函数 + 调用 `operator delete`             | 只释放内存，不调用析构函数                         |
| **类型安全**          | 类型安全（自动计算大小，无需强转）                | 非类型安全（需 `(int*)malloc` 强转）               |
| **失败处理**          | 默认抛出 `std::bad_alloc` 异常                    | 返回 `NULL`（需手动判断）                          |
| **构造函数/析构函数** | **自动调用**                                      | **绝不调用**（对象不完整）                         |
| **数组版本**          | `new[] / delete[]`（必须配对）                    | `malloc + 手动计算大小`（无专门数组版）            |
| **重载方式**          | 可全局/类内重载 `operator new/delete`             | 无法重载                                           |
| **性能**              | 稍慢（多一次构造/析构调用）                       | 更快（纯裸内存）                                   |
| **是否可自定义**      | 支持 placement new、not throw、aligned_new        | 不支持                                             |
| **C++标准要求**       | C++专有                                           | C语言遗留，C++兼容                                 |
| **现代推荐**          | 强烈推荐（配合智能指针）                          | 仅用于底层库或C接口                                |  
### 详细区别详解  
构造函数 & 析构函数  
``` cpp
class Test {
public:
    Test()  { std::cout << "构造\n"; }
    ~Test() { std::cout << "析构\n"; }
};

Test* p1 = new Test;     // 输出：构造
delete p1;               // 输出：析构

Test* p2 = (Test*)malloc(sizeof(Test));  // 无任何输出！
free(p2);                // 依然无任何输出！→ 内存泄漏 + 资源未释放！
```
异常安全  
``` cpp
// new 方式（推荐）
try {
    int* p = new int[1000000000];  // 可能抛 bad_alloc
} catch (const std::bad_alloc&) { ... }

// malloc 方式（老式）
int* p = (int*)malloc(1000000000 * sizeof(int));
if (p == nullptr) { /* 处理 */ }
```  
数组的正确与错误用法  
``` cpp
// 正确
std::string* arr = new std::string[5];  // 调用5次构造函数
delete[] arr;                           // 调用5次析构函数

// 错误示范（致命！）
std::string* arr = new std::string[5];
delete arr;          // 只调用1次析构 → 内存泄漏 + 未定义行为

// malloc 数组（无构造）
std::string* arr = (std::string*)malloc(5 * sizeof(std::string));
```  
重载与自定义  
``` cpp
// 全局重载 new/delete
void* operator new(size_t size) {
    std::cout << "全局 new 重载\n";
    return malloc(size);
}
void operator delete(void* p) noexcept {
    std::cout << "全局 delete 重载\n";
    free(p);
}

// 类内重载（最常见）
class MyClass {
public:
    void* operator new(size_t size) { ... }
    void operator delete(void* p) { ... }
};
```  
placement new  
``` cpp
char buffer[sizeof(Test)];
Test* p = new(buffer) Test;   // 在已有内存上构造（不分配内存）
p->~Test();                   // 必须手动调用析构（delete 不能用！）
```
底层实现原理  
new：  
调用 operator new(size)（可被重载）  
如果成功，调用构造函数  
失败抛异常（可加 new(std::nothrow) 不抛）  
  
delete：  
调用析构函数  
调用 operator delete(ptr)  
  
malloc/free：  
直接调用底层分配器（glibc ptmalloc / tcmalloc / jemalloc 等），无构造析构。  

最常被问的10个问题  
new 和 malloc 最大区别？  
→ “new 调用构造+析构，malloc 不；new 是运算符可重载，malloc 是函数。”  
为什么不能用 free 释放 new 的内存？  
→ “因为 new 可能额外记录了数组长度等信息，free 不会调用析构函数，导致资源泄漏。”  
delete 和 delete[] 区别？  
→ “delete 单个对象，delete[] 数组（会调用多次析构 + 读取数组长度）。”  
new int[10] 实际分配了多少内存？  
→ “比 40 字节多一点（通常多 4~8 字节存数组长度）。”  
new(std::nothrow) 是什么？  
→ “不抛异常版本，失败返回 nullptr，和 malloc 行为一致。”  
智能指针和 new 的关系？  
→ “std::unique_ptr、std::shared_ptr 底层依然用 new，但自动 delete，防止泄漏。”  
C++17/20 新增了什么？  
→ std::launder、std::aligned_alloc、operator new 对齐版本、std::destroy_at 等。  
malloc(0) 返回什么？  
→ “实现定义，可能返回 nullptr 或非 nullptr（但不能解引用）。”  
为什么库函数里常用 malloc？  
→ “因为要兼容 C，且不需要对象语义。”  
如何自己实现一个简易 new/delete？  
``` cpp
#include <iostream>
#include <new>       // ::operator new + placement new
#include <utility>   // std::forward

// ====================== 模拟 new（正确版 + 日志） ======================
template <typename T, typename... Args>
T* myNew(Args&&... args) {
    std::cout << "[myNew] 第1步: ::operator new 只分配内存（" << sizeof(T) << "字节），不构造\n";
    
    void* ptr = ::operator new(sizeof(T));   // 纯分配
    
    try {
        std::cout << "[myNew] 第2步: placement new 调用构造函数\n";
        return new (ptr) T(std::forward<Args>(args)...);
    } catch (...) {
        ::operator delete(ptr);
        throw;
    }
}

// ====================== 模拟 delete ======================
template <typename T>
void myDelete(T* ptr) {
    if (!ptr) return;
    std::cout << "[myDelete] 调用析构函数\n";
    ptr->~T();
    std::cout << "[myDelete] ::operator delete 释放内存\n";
    ::operator delete(ptr);
}

// ====================== 测试类 ======================
class MyClass {
public:
    MyClass(int v) : val(v) {
        std::cout << "【Constructor】val = " << val << std::endl;
    }
    ~MyClass() {
        std::cout << "【Destructor】val = " << val << std::endl;
    }
    int val;
};

int main() {
    std::cout << "=== 开始测试 ===\n";
    MyClass* obj = myNew<MyClass>(42);
    std::cout << "对象已创建，val = " << obj->val << "\n\n";
    
    myDelete(obj);
    return 0;
}

```

## 内存对齐  
### 1.为什么需要内存对齐？  
CPU 访问内存不是按字节，而是按字长（Word）读取：  
  
32位系统：一次读 4 字节  
64位系统：一次读 8 字节  
  
不对齐会导致：  
  
多次内存访问（性能下降 2~4 倍）  
跨缓存行（Cache Line，通常 64 字节）→ 伪共享（False Sharing）  
硬件异常（某些架构如 ARM 会直接崩溃）  
  
Padding：编译器自动在结构体成员之间插入无用字节，让每个成员的起始地址都是其对齐倍数。  
### 2. 对齐规则（C++ 标准 + 编译器实现）  
  
基本类型对齐值 = sizeof(类型)（或平台字长）  
char / bool：1 字节  
short：2 字节  
int / float：4 字节（32/64位通常都是 4）  
long / double / 指针：8 字节（64位）  
  
结构体成员对齐：每个成员的偏移量（offset）必须是自身对齐值的整数倍。  
结构体整体对齐：sizeof(struct) 是最大成员对齐值的整数倍（方便数组）。  
Padding 插入位置：  
成员之间（前置 padding）  
结构体末尾（尾部 padding）  
### 3.控制对齐的方式  
| 方式                          | 作用                          | 示例                                              | 备注                              |
|-------------------------------|-------------------------------|---------------------------------------------------|-----------------------------------|
| **默认**                      | 自然对齐（编译器默认行为）    | `struct S { ... };`                               | 推荐，大多数场景直接使用          |
| `#pragma pack(n)`             | 设置最大对齐值为 n            | `#pragma pack(1)` → 无 padding                    | 仅 MSVC / GCC / Clang 支持        |
| `__attribute__((packed))`     | 完全取消所有 padding          | `struct __attribute__((packed)) S {}`             | GCC / Clang 专有                  |
| `alignas(n)`                  | 强制结构体/成员对齐到 n 字节  | `struct alignas(16) S {};`<br>`alignas(64) int x;` | **C++11 标准**，最推荐的现代写法  |
| `alignof(T)`                  | 查询类型的对齐值              | `static_assert(alignof(int) == 4);`               | **C++11 标准**，编译期常量        |
### 4.其他知识点  
数组对齐：struct S arr[10]; → 每个元素按 struct 对齐，整体大小是 sizeof(S) * 10  
继承与虚函数：  
虚表指针（vptr）通常 8 字节，对齐 8  
基类 padding 会影响派生类  
C++17+ 新特性：  
std::aligned_storage  
std::align  
std::hardware_destructive_interference_size（64 字节）  
### 5.False Sharing  
现代 CPU Cache Line 大小通常是 64 字节（L1/L2/L3 缓存的最小单位）。  
当一个核心修改某个变量时，整个 64 字节缓存行会被标记为 Dirty（脏）。  
其他核心如果也要访问同一个缓存行里的任意字节，就会触发 Cache Invalidation（失效） + 跨核同步。  
结果：即使变量完全独立，也会像“真共享”一样互相干扰。  
真实代码对比  
有 False Sharing（性能极差）  
``` cpp
struct BadCounter {
    int counter1 = 0;
    int counter2 = 0;
};

BadCounter g;

void thread1() {
    for (long i = 0; i < 100000000; ++i) g.counter1++;
}

void thread2() {
    for (long i = 0; i < 100000000; ++i) g.counter2++;
}
```
解决：手动 Padding  
``` cpp
struct GoodCounter {
    int counter1 = 0;
    char padding[60];     // 手动补到 64 字节
    int counter2 = 0;
};
```
现代 C++17+ 最佳写法
``` cpp
#include <atomic>
#include <new>   // std::hardware_destructive_interference_size

struct PerfectCounter {
    alignas(std::hardware_destructive_interference_size) std::atomic<int> counter1{0};
    alignas(std::hardware_destructive_interference_size) std::atomic<int> counter2{0};
};
```
std::hardware_destructive_interference_size 就是当前平台真实的 Cache Line 大小（通常 64 或 128）
