# 内存模型   
## 栈/堆/全局/常量区/代码区  
<pre>
高地址  
┌────────────────────┐  
│       栈区 (Stack) │ ← 向下增长（函数调用）  
├────────────────────┤  
│       堆区 (Heap)  │ ← 向上增长（new/malloc）  
├────────────────────┤  
│ 未初始化数据 (BSS) │  
├────────────────────┤  
│ 已初始化数据 (Data)│  
├────────────────────┤  
│     常量区 (ROData)│  
├────────────────────┤  
│     代码区 (Text)  │ ← 最低地址  
└────────────────────┘  
低地址  
</pre>  
### 代码区（Code Segment / Text Segment）  
存放内容：程序编译后的机器指令（可执行代码）  
特点：  
只读（防止被修改）  
可共享（多个进程可以映射同一份代码，节省内存）  
大小在编译期确定，运行时不可变  
  
生命周期：整个程序运行期间  
示例：你写的 main()、func() 等函数体都在这里  
“代码区存放二进制指令，不允许修改，否则会触发段错误（Segmentation Fault）”  
  
### 常量区（Constant Segment / ROData）  
存放内容：  
字符串字面量（如 "hello world"） 
const全局/静态变量  
constexpr 常量（C++11起）  
  
特点：  
只读（尝试修改会段错误）  
编译期确定  
属于只读数据段，常与代码区放在一起  

生命周期：整个程序  
示例：  
``` cpp
const int GLOBAL_CONST = 100;     // 常量区
constexpr double PI = 3.14;       // 常量区
const char* str = "hello";        // "hello" 在常量区，str指针在栈/全局
```
### 全局区 / 静态存储区（Data Segment + BSS）  
细分为两部分：  
已初始化数据段（Data）：显式初始化过的全局/静态变量  
未初始化数据段（BSS）：未初始化的全局/静态变量（系统自动清0）  
  
存放内容：  
全局变量（int g = 10;）  
静态变量（static int s; 无论在函数内还是外）  
static 局部变量（只初始化一次）  
  
特点：  
程序启动前分配，程序结束时释放  
生命周期 = 整个程序  

示例：  
``` cpp
int global_var = 10;          // Data段
static int static_global = 20;// Data段
int uninit_global;            // BSS段（值为0）

void func() {
    static int count = 0;     // 静态局部变量，只初始化一次，放在全局区
    count++;
}
```

### 堆区（Heap）  
存放内容：程序员动态申请的内存  
分配方式：new / malloc / calloc / realloc
特点：  
向上增长（地址越来越大）  
大小几乎不受限制（受虚拟内存和系统限制）  
手动管理（delete / free）  
分配/释放较慢（涉及系统调用）  
  
生命周期：从 new 到 delete，可跨函数  
示例：    
``` cpp
int* p = new int[1000000];  // 堆
delete[] p;
```   
### 栈区（Stack）  
存放内容：  
局部变量  
函数参数  
返回地址、调用者栈帧（ebp、esp）  
  
分配方式：自动（进入函数时分配，退出时释放）  
特点：  
向下增长（地址越来越小）  
大小有限（Linux默认8MB，Windows默认1MB，可通过编译器参数修改）  
分配/释放极快（只需移动栈指针）  
  
生命周期：函数作用域结束自动释放  
示例：  
``` cpp
void func(int param) {      // param 在栈
    int local = 10;         // local 在栈
    int arr[1000000];       // 太大 → 栈溢出！
}
```  
### 最常被问的10个问题  
栈和堆的区别？  
→ 见上表 + “栈自动管理、连续、快；堆手动、易碎片、慢”  
为什么栈比堆快？  
→ 栈只需移动esp指针；堆需要系统调用 + 找空闲块（可能用伙伴系统/空闲链表）  
const全局变量和const局部变量分别在哪？  
→ 全局const在常量区；局部const通常在栈（除非被优化到常量区）  
static变量在哪？什么时候初始化？  
→ 全局区；程序启动前（全局静态）或第一次调用时（局部静态）  
new 和 malloc 的区别？  
→ new是运算符（调用构造函数）；malloc是函数（只分配内存）  
字符串 "hello" 在哪？  
→ 常量区（只读）  
全局变量未初始化值是多少？  
→ 0（BSS段特性）  
如何防止栈溢出？  
→ 避免大数组、减少递归深度、用堆代替大局部变量  
内存泄漏如何检测？  
→ valgrind、AddressSanitizer、CRT调试堆  
C++11之后新增的内存相关关键字？  
→ thread_local（线程局部存储）、constexpr、std::aligned_storage 等  
  
## new/delete vs malloc/free  
| 对比维度              | new / delete                                      | malloc / free                                      |
|-----------------------|---------------------------------------------------|----------------------------------------------------|
| **本质**              | **运算符**（可重载）                              | **库函数**（不能重载）                             |
| **分配内存**          | 调用 `operator new` 分配 + 调用构造函数           | 只调用底层分配器（通常是 `malloc`）                |
| **释放内存**          | 调用析构函数 + 调用 `operator delete`             | 只释放内存，不调用析构函数                         |
| **类型安全**          | 类型安全（自动计算大小，无需强转）                | 非类型安全（需 `(int*)malloc` 强转）               |
| **失败处理**          | 默认抛出 `std::bad_alloc` 异常                    | 返回 `NULL`（需手动判断）                          |
| **构造函数/析构函数** | **自动调用**                                      | **绝不调用**（对象不完整）                         |
| **数组版本**          | `new[] / delete[]`（必须配对）                    | `malloc + 手动计算大小`（无专门数组版）            |
| **重载方式**          | 可全局/类内重载 `operator new/delete`             | 无法重载                                           |
| **性能**              | 稍慢（多一次构造/析构调用）                       | 更快（纯裸内存）                                   |
| **是否可自定义**      | 支持 placement new、not throw、aligned_new        | 不支持                                             |
| **C++标准要求**       | C++专有                                           | C语言遗留，C++兼容                                 |
| **现代推荐**          | 强烈推荐（配合智能指针）                          | 仅用于底层库或C接口                                |  
### 详细区别详解  
构造函数 & 析构函数  
``` cpp
class Test {
public:
    Test()  { std::cout << "构造\n"; }
    ~Test() { std::cout << "析构\n"; }
};

Test* p1 = new Test;     // 输出：构造
delete p1;               // 输出：析构

Test* p2 = (Test*)malloc(sizeof(Test));  // 无任何输出！
free(p2);                // 依然无任何输出！→ 内存泄漏 + 资源未释放！
```
异常安全  
``` cpp
// new 方式（推荐）
try {
    int* p = new int[1000000000];  // 可能抛 bad_alloc
} catch (const std::bad_alloc&) { ... }

// malloc 方式（老式）
int* p = (int*)malloc(1000000000 * sizeof(int));
if (p == nullptr) { /* 处理 */ }
```  
数组的正确与错误用法  
``` cpp
// 正确
std::string* arr = new std::string[5];  // 调用5次构造函数
delete[] arr;                           // 调用5次析构函数

// 错误示范（致命！）
std::string* arr = new std::string[5];
delete arr;          // 只调用1次析构 → 内存泄漏 + 未定义行为

// malloc 数组（无构造）
std::string* arr = (std::string*)malloc(5 * sizeof(std::string));
```  
重载与自定义  
``` cpp
// 全局重载 new/delete
void* operator new(size_t size) {
    std::cout << "全局 new 重载\n";
    return malloc(size);
}
void operator delete(void* p) noexcept {
    std::cout << "全局 delete 重载\n";
    free(p);
}

// 类内重载（最常见）
class MyClass {
public:
    void* operator new(size_t size) { ... }
    void operator delete(void* p) { ... }
};
```  
placement new  
``` cpp
char buffer[sizeof(Test)];
Test* p = new(buffer) Test;   // 在已有内存上构造（不分配内存）
p->~Test();                   // 必须手动调用析构（delete 不能用！）
```
底层实现原理  
new：  
调用 operator new(size)（可被重载）  
如果成功，调用构造函数  
失败抛异常（可加 new(std::nothrow) 不抛）  
  
delete：  
调用析构函数  
调用 operator delete(ptr)  
  
malloc/free：  
直接调用底层分配器（glibc ptmalloc / tcmalloc / jemalloc 等），无构造析构。  

最常被问的10个问题  
new 和 malloc 最大区别？  
→ “new 调用构造+析构，malloc 不；new 是运算符可重载，malloc 是函数。”  
为什么不能用 free 释放 new 的内存？  
→ “因为 new 可能额外记录了数组长度等信息，free 不会调用析构函数，导致资源泄漏。”  
delete 和 delete[] 区别？  
→ “delete 单个对象，delete[] 数组（会调用多次析构 + 读取数组长度）。”  
new int[10] 实际分配了多少内存？  
→ “比 40 字节多一点（通常多 4~8 字节存数组长度）。”  
new(std::nothrow) 是什么？  
→ “不抛异常版本，失败返回 nullptr，和 malloc 行为一致。”  
智能指针和 new 的关系？  
→ “std::unique_ptr、std::shared_ptr 底层依然用 new，但自动 delete，防止泄漏。”  
C++17/20 新增了什么？  
→ std::launder、std::aligned_alloc、operator new 对齐版本、std::destroy_at 等。  
malloc(0) 返回什么？  
→ “实现定义，可能返回 nullptr 或非 nullptr（但不能解引用）。”  
为什么库函数里常用 malloc？  
→ “因为要兼容 C，且不需要对象语义。”  
如何自己实现一个简易 new/delete？  
``` cpp
#include <iostream>
#include <new>       // ::operator new + placement new
#include <utility>   // std::forward

// ====================== 模拟 new（正确版 + 日志） ======================
template <typename T, typename... Args>
T* myNew(Args&&... args) {
    std::cout << "[myNew] 第1步: ::operator new 只分配内存（" << sizeof(T) << "字节），不构造\n";
    
    void* ptr = ::operator new(sizeof(T));   // 纯分配
    
    try {
        std::cout << "[myNew] 第2步: placement new 调用构造函数\n";
        return new (ptr) T(std::forward<Args>(args)...);
    } catch (...) {
        ::operator delete(ptr);
        throw;
    }
}

// ====================== 模拟 delete ======================
template <typename T>
void myDelete(T* ptr) {
    if (!ptr) return;
    std::cout << "[myDelete] 调用析构函数\n";
    ptr->~T();
    std::cout << "[myDelete] ::operator delete 释放内存\n";
    ::operator delete(ptr);
}

// ====================== 测试类 ======================
class MyClass {
public:
    MyClass(int v) : val(v) {
        std::cout << "【Constructor】val = " << val << std::endl;
    }
    ~MyClass() {
        std::cout << "【Destructor】val = " << val << std::endl;
    }
    int val;
};

int main() {
    std::cout << "=== 开始测试 ===\n";
    MyClass* obj = myNew<MyClass>(42);
    std::cout << "对象已创建，val = " << obj->val << "\n\n";
    
    myDelete(obj);
    return 0;
}

```

## 内存对齐  
### 1.为什么需要内存对齐？  
CPU 访问内存不是按字节，而是按字长（Word）读取：  
  
32位系统：一次读 4 字节  
64位系统：一次读 8 字节  
  
不对齐会导致：  
  
多次内存访问（性能下降 2~4 倍）  
跨缓存行（Cache Line，通常 64 字节）→ 伪共享（False Sharing）  
硬件异常（某些架构如 ARM 会直接崩溃）  
  
Padding：编译器自动在结构体成员之间插入无用字节，让每个成员的起始地址都是其对齐倍数。  
### 2. 对齐规则（C++ 标准 + 编译器实现）  
  
基本类型对齐值 = sizeof(类型)（或平台字长）  
char / bool：1 字节  
short：2 字节  
int / float：4 字节（32/64位通常都是 4）  
long / double / 指针：8 字节（64位）  
  
结构体成员对齐：每个成员的偏移量（offset）必须是自身对齐值的整数倍。  
结构体整体对齐：sizeof(struct) 是最大成员对齐值的整数倍（方便数组）。  
Padding 插入位置：  
成员之间（前置 padding）  
结构体末尾（尾部 padding）  
### 3.控制对齐的方式  
| 方式                          | 作用                          | 示例                                              | 备注                              |
|-------------------------------|-------------------------------|---------------------------------------------------|-----------------------------------|
| **默认**                      | 自然对齐（编译器默认行为）    | `struct S { ... };`                               | 推荐，大多数场景直接使用          |
| `#pragma pack(n)`             | 设置最大对齐值为 n            | `#pragma pack(1)` → 无 padding                    | 仅 MSVC / GCC / Clang 支持        |
| `__attribute__((packed))`     | 完全取消所有 padding          | `struct __attribute__((packed)) S {}`             | GCC / Clang 专有                  |
| `alignas(n)`                  | 强制结构体/成员对齐到 n 字节  | `struct alignas(16) S {};`<br>`alignas(64) int x;` | **C++11 标准**，最推荐的现代写法  |
| `alignof(T)`                  | 查询类型的对齐值              | `static_assert(alignof(int) == 4);`               | **C++11 标准**，编译期常量        |
### 4.其他知识点  
数组对齐：struct S arr[10]; → 每个元素按 struct 对齐，整体大小是 sizeof(S) * 10  
继承与虚函数：  
虚表指针（vptr）通常 8 字节，对齐 8  
基类 padding 会影响派生类  
C++17+ 新特性：  
std::aligned_storage  
std::align  
std::hardware_destructive_interference_size（64 字节）  
### 5.False Sharing  
现代 CPU Cache Line 大小通常是 64 字节（L1/L2/L3 缓存的最小单位）。  
当一个核心修改某个变量时，整个 64 字节缓存行会被标记为 Dirty（脏）。  
其他核心如果也要访问同一个缓存行里的任意字节，就会触发 Cache Invalidation（失效） + 跨核同步。  
结果：即使变量完全独立，也会像“真共享”一样互相干扰。  
真实代码对比  
有 False Sharing（性能极差）  
``` cpp
struct BadCounter {
    int counter1 = 0;
    int counter2 = 0;
};

BadCounter g;

void thread1() {
    for (long i = 0; i < 100000000; ++i) g.counter1++;
}

void thread2() {
    for (long i = 0; i < 100000000; ++i) g.counter2++;
}
```
解决：手动 Padding  
``` cpp
struct GoodCounter {
    int counter1 = 0;
    char padding[60];     // 手动补到 64 字节
    int counter2 = 0;
};
```
现代 C++17+ 最佳写法
``` cpp
#include <atomic>
#include <new>   // std::hardware_destructive_interference_size

struct PerfectCounter {
    alignas(std::hardware_destructive_interference_size) std::atomic<int> counter1{0};
    alignas(std::hardware_destructive_interference_size) std::atomic<int> counter2{0};
};
```
std::hardware_destructive_interference_size 就是当前平台真实的 Cache Line 大小（通常 64 或 128）  

## 智能指针  
### 1. 智能指针核心思想  
智能指针 = RAII + 自动资源管理  
利用构造函数申请资源、析构函数释放资源，彻底杜绝手动 new/delete 导致的内存泄漏、野指针、double delete 等问题。  
### 2. 三种智能指针对比表
| 对比维度           | unique_ptr                          | shared_ptr                              | weak_ptr                              |
|--------------------|-------------------------------------|-----------------------------------------|---------------------------------------|
| **所有权**         | 独占（Exclusive）                   | 共享（Shared）                          | 非拥有（Observer）                    |
| **能否拷贝**       | 不可以（只能移动）                  | 可以（引用计数+1）                      | 可以（不影响计数）                    |
| **能否多线程共享** | 不可（非线程安全）                  | 可以（引用计数原子操作）                | 可以                                  |
| **内存开销**       | 几乎无（只存裸指针）                | 控制块（ref count + weak count）≈ 2个指针 | 只存控制块指针（极小）                |
| **循环引用问题**   | 无（独占不会循环）                  | 会导致内存泄漏                          | 专门用于打破循环引用                  |
| **释放时机**       | 离开作用域或被 move                 | 引用计数归0                             | 不负责释放，仅观察                    |
| **推荐创建方式**   | `std::make_unique<T>(args)`         | `std::make_shared<T>(args)`             | `std::weak_ptr<T> w = s;`             |
| **典型场景**       | 独占资源、工厂返回、Pimpl 惯用法    | 共享数据结构、缓存、观察者模式          | 解决 shared_ptr 循环引用、缓存避免悬垂 |  
### 3. unique_ptr 详解  
特点：独占所有权，不可拷贝，只能移动（std::move）。  
大小：通常等于裸指针（8字节，64位）。  
自定义删除器：  
``` cpp
std::unique_ptr<FILE, decltype(&fclose)> fp(fopen("test.txt", "r"), fclose);
```
现代创建方式（C++14+）：  
```cpp
auto p = std::make_unique<MyClass>(10, "hello");
```  
Pimpl 惯用法（隐藏实现，减少头文件依赖）：  
``` cpp
// MyClass.h
class MyClass {
    struct Impl;
    std::unique_ptr<Impl> pImpl;
public:
    MyClass();
    ~MyClass();  // 必须在 .cpp 中定义
};
```  
### 4. shared_ptr 详解  
原理：引用计数（strong reference count）+ 控制块（control block）。  
make_shared 优势（必背！）：  
一次分配（对象 + 控制块），减少 1 次 malloc。  
异常安全（new 失败不会泄漏控制块）。  
更好的缓存局部性。  
``` cpp
auto p1 = std::make_shared<int>(42);   // 推荐
auto p2 = std::shared_ptr<int>(new int(42));  // 反模式（两次分配）
```
### 5. weak_ptr 详解 + 循环引用彻底解决5. weak_ptr 详解 + 循环引用彻底解决  
weak_ptr 不增加引用计数，只“观察”对象是否存活。  
循环引用导致内存泄漏（经典例子）  
``` cpp
struct A {
    std::shared_ptr<B> pb;
};
struct B {
    std::shared_ptr<A> pa;
};

auto a = std::make_shared<A>();
auto b = std::make_shared<B>();
a->pb = b;
b->pa = a;   // 循环引用！a 和 b 永远不会被释放
```
用 weak_ptr 打破循环  
``` cpp
#include <iostream>
#include <memory>

struct Child;

struct Parent {
    std::shared_ptr<Child> child;
    ~Parent() { std::cout << "Parent 析构\n"; }
};

struct Child {
    std::weak_ptr<Parent> parent;   // 关键：weak_ptr
    ~Child() { std::cout << "Child 析构\n"; }
};

int main() {
    auto p = std::make_shared<Parent>();
    auto c = std::make_shared<Child>();
    p->child = c;
    c->parent = p;     // weak_ptr 不增加计数

    std::cout << "p use_count: " << p.use_count() << "\n";  // 1
    // 离开作用域时自动释放，无泄漏
}
```
### 6. 线程安全详解  
| 操作场景                     | 是否线程安全          | 说明                                                                 |
|------------------------------|-----------------------|----------------------------------------------------------------------|
| **多个线程同时读同一个 shared_ptr** | 安全                  | 引用计数原子操作                                                     |
| **多个线程同时拷贝 shared_ptr** | 安全                  | 引用计数原子递增                                                     |
| **一个线程写 + 其他线程读**   | 不安全                | 需加 mutex 保护对象本身                                              |
| **unique_ptr 跨线程传递**     | 不安全（只能 move）   | 移动后原指针失效                                                     |
| **shared_ptr 作为函数参数**   | 推荐按值传递          | 自动拷贝，线程安全                                                   |
| **shared_ptr 作为成员变量**   | 需外部保护            | 对象本身不是线程安全的                                               |
| **weak_ptr::lock()**          | 线程安全              | 原子操作                                                             |  
关键结论  
shared_ptr 的引用计数是线程安全的（内部使用 atomic）。  
被管理的对象本身不一定是线程安全的（你要自己加锁）。  
推荐：std::atomic<std::shared_ptr<T>>（C++20）或手动 mutex。  
### 7. 现代 C++ 最佳实践  
```cpp
// 1. 永远优先 make_xxx
auto p = std::make_unique<MyClass>(args...);
auto s = std::make_shared<MyClass>(args...);

// 2. 传递时：
void func(std::shared_ptr<T> sp);     // 按值传递（推荐）
void func(const std::shared_ptr<T>& sp); // 只读时可引用

// 3. 自定义删除器 + 数组
std::unique_ptr<int[], decltype(&free)> arr((int*)malloc(100*sizeof(int)), free);

// 4. C++17+ enable_shared_from_this
class Widget : public std::enable_shared_from_this<Widget> {
public:
    std::shared_ptr<Widget> getShared() { return shared_from_this(); }
};
```
### 最常被问的 10 个问题  
unique_ptr 能不能拷贝？ → 不可以，只能 std::move。  
shared_ptr 循环引用怎么解决？ → 用 weak_ptr 打破环（经典例子必背）。  
make_shared 比 new 有什么优势？ → 一次分配、异常安全、缓存友好。  
weak_ptr::expired() 和 lock() 区别？ → expired() 只检查，lock() 返回 shared_ptr（可能为空）。  
shared_ptr 线程安全吗？ → 引用计数安全，对象本身不安全。  
unique_ptr 能放进 vector 吗？ → 可以（支持移动）。  
自定义删除器如何写？ → std::unique_ptr<T, Deleter>。  
shared_ptr 析构时做了什么？ → 计数-1，归0则释放对象。  
enable_shared_from_this 作用？ → 在类内部安全获得 shared_ptr。  
C++20 有什么新特性？ → std::atomic<std::shared_ptr<T>>、std::make_shared_for_overwrite 等。  

## RAII 原理详解  
### 1. RAII 是什么？  
RAII 全称 Resource Acquisition Is Initialization  
资源获取即初始化  
资源释放即对象析构  
核心思想：  
在对象的构造函数中获取资源（new、malloc、open file、lock mutex 等），  
在对象的析构函数中释放资源（delete、free、close、unlock 等）。  
这样，资源的生命周期就严格绑定到对象的生命周期，离开作用域自动释放。  
### 2. RAII 的四大核心优势  
| 优势               | 说明                                                                 | 对比传统手动管理的问题                     |
|--------------------|----------------------------------------------------------------------|--------------------------------------------|
| **异常安全**       | 即使抛出异常，栈展开时也会自动调用析构函数释放资源                   | 异常后容易忘记 free / close / unlock      |
| **确定性释放**     | 资源释放时机明确（作用域结束 / 对象销毁时），不依赖程序员手动调用   | 忘记 delete / close 导致泄漏               |
| **代码简洁**       | 无需写成堆的 try-finally / goto cleanup                              | 代码冗长、容易出错                         |
| **线程安全潜力**   | 配合智能指针 / lock_guard 等，天然支持 RAII 风格的资源管理           | 手动管理容易出现 race condition            |  
### 3. RAII 的经典实现方式对比  
| 资源类型       | 传统 C 风格（易出错）                          | RAII 风格（现代 C++ 推荐）                          | 备注                              |
|----------------|------------------------------------------------|-----------------------------------------------------|-----------------------------------|
| 动态内存       | `int* p = new int; ... delete p;`              | `std::unique_ptr<int> p = std::make_unique<int>();` | 自动释放，无需 delete             |
| 文件句柄       | `FILE* f = fopen(...); ... fclose(f);`         | `std::ofstream file("log.txt");`                    | 析构时自动 close                  |
| 互斥锁         | `pthread_mutex_lock(&m); ... pthread_mutex_unlock(&m);` | `std::lock_guard<std::mutex> lk(m);`                | 离开作用域自动解锁                |
| 数据库连接     | `conn = db_connect(); ... db_disconnect(conn);` | 自定义 RAII 类 `DbConnection conn;`                | 析构时断开连接                    |
| 网络 socket    | `int sock = socket(...); ... close(sock);`     | 自定义 RAII wrapper                                 | 析构时 close                      |  
### 4. 经典 RAII 示例代码  
示例 1：文件 RAII 封装  
``` cpp
class File {
private:
    FILE* fp;
public:
    explicit File(const char* filename, const char* mode = "r") {
        fp = fopen(filename, mode);
        if (!fp) {
            throw std::runtime_error("fopen failed");
        }
    }

    ~File() {
        if (fp) {
            fclose(fp);
            fp = nullptr;
        }
    }

    // 禁止拷贝（或实现拷贝语义）
    File(const File&) = delete;
    File& operator=(const File&) = delete;

    // 提供访问接口
    FILE* get() const { return fp; }
    size_t read(void* buf, size_t size) { return fread(buf, 1, size, fp); }
};

// 使用方式（异常安全）
void process_log() {
    File log("app.log", "a");
    fprintf(log.get(), "Error occurred at %s\n", __TIME__);
    // 离开作用域 → 自动 fclose，无需手动 close
}
```  
示例 2：lock_guard（标准库经典 RAII）  
``` cpp
std::mutex m;

void safe_increment(int& counter) {
    std::lock_guard<std::mutex> guard(m);  // 构造时 lock
    ++counter;
    // 离开作用域 → 析构时 unlock
}
```
### 5. RAII 与异常安全  
“异常发生时，栈上所有局部对象的析构函数都会被调用”  
这正是 RAII 能保证异常安全的根本原因。  
  
三种异常安全级别：  
基本异常安全：资源不泄漏，但状态可能不一致  
强异常安全：要么成功，要么什么都不发生（回滚）  
无异常抛出（not throw）：最强保证  
  
RAII 天然提供基本异常安全，配合事务 / copy-and-swap 等可实现强异常安全。  

### 6. 现代 C++ 中 RAII 的最佳实践  
优先使用标准库 RAII 工具：unique_ptr、shared_ptr、lock_guard、unique_lock、scoped_lock、std::jthread、std::ofstream 等  
需要自定义 RAII 类时：  
实现析构函数释放资源  
禁用拷贝（或实现移动语义）  
提供 release() / reset() 等接口  
析构函数标记 noexcept
